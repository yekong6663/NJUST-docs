## 1. 从浏览器输入网址到页面显示，描述完整的数据流及各层协议

1. **应用层**
   - 浏览器（HTTP客户端）构造 HTTP GET 请求报文，请求访问 `http://www.example.com` 的默认页面。
   - 同时，浏览器需要将域名 `www.example.com` 转换为 IP 地址，因此调用 DNS 协议。
   - DNS 客户端向本地域名服务器发送 UDP 查询报文（递归查询），最终获得目的 IP 地址（如 `93.184.216.34`）。
   - HTTP 请求报文作为应用层数据单元（**报文**）向下交付给运输层。

2. **运输层**
   - HTTP 协议使用 TCP 的熟知端口 80。
   - 首先进行**三次握手**建立 TCP 连接：客户端发送 SYN（seq=x），服务器回复 SYN+ACK（seq=y, ack=x+1），客户端回复 ACK（ack=y+1）。
   - 连接建立后，HTTP 请求报文被封装成 TCP **报文段**，添加源端口（临时端口）、目的端口（80）、序号、确认号、窗口等字段。
   - TCP 提供可靠、面向连接的服务，确保数据不丢失、按序到达。

3. **网络层**
   - TCP 报文段被封装成 IP **数据报**（也称分组）。
   - IP 数据报头部包含源 IP 地址、目的 IP 地址（`93.184.216.34`）、TTL、协议字段（6 表示 TCP）等。
   - 源主机根据路由表确定下一跳路由器（通常为默认网关），可能需要进行 IP 分片（若数据报长度超过 MTU 1500 字节）。

4. **数据链路层**
   - IP 数据报被封装成以太网**帧**。
   - 源主机需要知道下一跳路由器的 MAC 地址，因此调用 **ARP 协议**：广播 ARP 请求（“谁是网关 IP？”），网关单播回复其 MAC 地址。
   - 帧头部填入源 MAC 地址、目的 MAC 地址（网关 MAC），类型字段设为 `0x0800`（表示 IP 数据报）。
   - 帧尾部添加 FCS（CRC 校验），用于检测传输错误。

5. **物理层**
   - 帧转换为比特流，经网卡、双绞线或光纤发送到物理介质上。
   - 物理层定义了电压、电缆接口、传输速率等。

6. **中间路由器转发**
   - 每台路由器接收到帧后，先在物理层接收比特，数据链路层解封装并校验 FCS（错误则丢弃）。
   - 网络层提取 IP 数据报，根据目的 IP 地址查找路由表，决定下一跳。
   - 重新封装成新的帧（MAC 地址更新为下一跳的 MAC），继续转发，直到到达目的主机所在网络。

7. **目的主机接收**
   - 目的主机从物理层逐层向上解封装。
   - 网络层将 IP 数据报交给运输层，TCP 重组报文段，发送 ACK 确认。
   - 运输层将 HTTP 请求报文交给应用层的 Web 服务器进程。
   - Web 服务器处理请求，返回 HTTP 响应（状态行、首部、数据）。
   - 响应报文经过同样的封装、传输过程回到浏览器。
   - 浏览器解析 HTML、渲染页面，最终显示给用户。

## 2. 比较集线器、网桥、交换机、路由器、网关



1. **集线器（Hub）**
   - **工作层次**：物理层。
   - **主要功能**：信号再生、整形，将接收到的比特转发到所有其他端口。
   - **冲突域**：所有端口属于同一个冲突域，同一时刻只能有一个站发送数据，否则碰撞。
   - **广播域**：所有端口属于同一个广播域。
   - **带宽共享**：共享总带宽，例如 10M 集线器连接 5 台主机，每台平均仅 2M。
   - **典型场景**：小型简单网络，现已基本被交换机取代。

2. **网桥（Bridge）**
   - **工作层次**：数据链路层。
   - **主要功能**：根据 MAC 地址表过滤/转发帧，能够连接两个相同类型的网段，隔离冲突。
   - **冲突域**：每个端口独立冲突域（因为帧不会转发回原端口）。
   - **广播域**：所有端口仍属于同一个广播域（广播帧会转发到所有端口）。
   - **自学习**：通过源 MAC 地址自动建立地址表。
   - **典型场景**：早期以太网扩展，现已由多端口网桥（即交换机）替代。

3. **交换机（Switch）**
   - **工作层次**：数据链路层（多端口网桥）。
   - **主要功能**：MAC 地址表自学习、帧过滤转发、全双工通信、并行转发。
   - **冲突域**：每个端口独立冲突域，无碰撞。
   - **广播域**：所有端口仍属于同一个广播域（除非划分 VLAN）。
   - **带宽优势**：每个端口独享带宽，例如 100M 交换机，8 个端口可同时以 100M 通信，总容量 800M。
   - **典型场景**：现代局域网核心设备。

4. **路由器（Router）**
   - **工作层次**：网络层。
   - **主要功能**：根据 IP 地址路由选择、转发 IP 数据报，连接不同网络，实现网络层互连。
   - **冲突域**：每个端口隔离冲突域。
   - **广播域**：每个端口隔离广播域（路由器默认不转发广播）。
   - **其他功能**：NAT、DHCP 服务器、防火墙、VPN 等。
   - **典型场景**：连接局域网与互联网，不同网段互连。

5. **网关（Gateway）**
   - **工作层次**：网络层以上（常指运输层或应用层）。
   - **主要功能**：协议转换，如 TCP/IP 与 SNA、AppleTalk 等不同协议栈之间的转换。
   - **与路由器关系**：在 TCP/IP 网络中，“网关”有时泛指路由器，但严格而言网关功能更强大。
   - **典型场景**：异构网络互连、物联网协议转换、应用层网关（如 HTTP 代理）。

## 3. 电路交换、报文交换、分组交换的优缺点及分组交换的两种服务



1. **电路交换**
   - **过程**：建立连接 → 通信 → 释放连接。
   - **优点**：时延小（通信期间独占资源）；实时性强；无失序问题；适用于模拟和数字信号；设备相对简单。
   - **缺点**：建立连接时间长；信道利用率低（资源始终占用，即使无数据传输）；不同类型终端难以互通；不适用于突发数据。

2. **报文交换**
   - **过程**：存储转发，以整个报文为单位。
   - **优点**：无需预先建立连接；信道共享，利用率高；可实现优先级调度。
   - **缺点**：时延非常大（整个报文存储转发）；需要较大的缓存空间；无法保证实时性。

3. **分组交换**
   - **过程**：将报文划分为固定长度的分组，存储转发。
   - **优点**：高效（动态分配带宽）；灵活（每个分组独立选路）；迅速（无需建立连接，分组短，存储快）；可靠（多路径，有差错控制机制）。
   - **缺点**：排队时延可能较大；控制信息（首部）带来开销；分组可能失序、丢失，需高层处理；需要拥塞控制机制。

4. **分组交换内部的两种服务方式**
   - **数据报服务（如 IP）**
     - 每个分组独立选择路由，不建立连接。
     - 分组可能不按序到达，网络不保证可靠交付。
     - 适合突发性、容忍失序的应用（如普通数据通信、Web 浏览）。
   - **虚电路服务（如 X.25、帧中继）**
     - 通信前建立逻辑连接（虚电路），所有分组沿同一路径顺序传送。
     - 网络可以提供可靠交付，分组按序到达。
     - 适合长时间、连续的数据流（如 VoIP、视频会议）。

## 4. TCP 和 UDP 封装对比及适用场景



1. **TCP 封装过程**
   - 应用层数据（如 HTTP 请求）被 TCP 协议封装成 **TCP 报文段**。
   - TCP 报文段首部长度 20~60 字节，包含源端口、目的端口、序号、确认号、数据偏移、保留、标志位（URG、ACK、PSH、RST、SYN、FIN）、窗口、校验和、紧急指针、选项等。
   - 校验和计算时需加上 12 字节的**伪首部**（包含源 IP、目的 IP、全0、协议号6、TCP 长度）。
   - 发送前需先通过**三次握手**建立连接，提供可靠传输、流量控制、拥塞控制。

2. **UDP 封装过程**
   - 应用层数据被 UDP 协议封装成 **UDP 用户数据报**。
   - UDP 首部固定 8 字节，包含源端口、目的端口、长度、校验和。
   - 校验和可选（IPv4 中可为 0），计算时也需加上 12 字节伪首部（协议号 17）。
   - UDP 无连接，不需要事先建立连接，不提供可靠交付、无拥塞控制。

3. **适用场景对比**
   - **TCP 适用的应用**
     - 要求可靠、按序、无丢失：HTTP/HTTPS（网页）、FTP（文件传输）、SMTP/POP3/IMAP（邮件）、Telnet/SSH（远程登录）。
   - **UDP 适用的应用**
     - 实时性强、可容忍少量丢包：DNS（域名查询，简单请求响应）、RTP（实时音视频）、VoIP（语音）、SNMP（网络管理）、TFTP（简单文件传输）、DHCP（动态配置）。
     - 广播/多播通信也使用 UDP。

## 5. 跨网络通信中 ARP、IP、路由协议、DNS 的协作



1. **DNS 的作用**
   - 用户使用域名访问，例如 `b.example.com`。
   - 主机 A 先向本地 DNS 服务器发起查询（UDP 53 端口），通过递归或迭代查询，获得目的主机 B 的 IP 地址（如 `192.168.1.5`）。

2. **IP 转发与路由表**
   - 主机 A 将目的 IP `192.168.1.5` 与自己子网掩码 `255.255.255.0` 比较，发现不在同一网络。
   - 因此主机 A 将 IP 数据报发送给默认网关（例如 `10.0.0.254`），网关路由器收到后，根据目的 IP 查找路由表决定下一跳。

3. **路由协议（RIP/OSPF/BGP）**
   - 在自治系统内部，路由器之间通过 **内部网关协议（IGP）** 如 RIP、OSPF 动态交换路由信息。
   - RIP 基于距离向量（跳数），每 30 秒交换完整路由表，适合小型网络。
   - OSPF 基于链路状态，使用 Dijkstra 算法，收敛快，适合大型网络。
   - 路由器根据协议学习的路由表，知道到达 `192.168.1.0/24` 网络的最佳下一跳。

4. **ARP 的作用**
   - 在每一段链路转发时，路由器需要知道下一跳 IP 对应的 MAC 地址。
   - 例如主机 A 发送帧给网关 `10.0.0.254`：A 广播 ARP 请求（“谁是 10.0.0.254？”），网关单播回复其 MAC 地址。
   - 后续每一跳路由器之间也使用 ARP 解析相邻路由器的 MAC。
   - ARP 仅在本地网络有效，不跨路由器。

5. **综合流程**
   - DNS 解析域名 → 得到目的 IP。
   - 源主机封装 IP 数据报，根据路由表找到下一跳网关。
   - ARP 解析网关 MAC，发送帧。
   - 网关路由器解封装，根据路由表（由路由协议建立）继续转发。
   - 重复直到到达目的网络，最终 ARP 解析目的主机 MAC，交付帧。
   - 所有协议协作完成跨网络通信。

## 6. 以太网交换机自学习过程及与集线器的性能对比



1. **交换机自学习算法步骤**
   - ① 交换机从某个端口收到一个帧，提取**源 MAC 地址**和**进入端口号**。
   - ② 在 MAC 地址表中查找该源 MAC：
     - 若不存在，则添加新条目（MAC、端口、有效时间 TTL）。
     - 若存在但端口不同，则更新为最新端口（因为主机可能移动）。
   - ③ 提取**目的 MAC 地址**，查找地址表：
     - 若找到且目的端口与进入端口不同，则向该端口转发帧。
     - 若找到但目的端口与进入端口相同，则丢弃帧（无需转发）。
     - 若未找到，则向**除进入端口外的所有端口**广播（洪泛）。
   - ④ 定期老化（删除）超时未更新的条目，避免表过大。

2. **交换机与集线器的性能对比**
   - **工作方式**：
     - 集线器：物理层，比特再生后向所有端口广播。
     - 交换机：数据链路层，根据 MAC 地址有选择地转发。
   - **冲突域**：
     - 集线器：所有端口属于同一冲突域，容易产生碰撞。
     - 交换机：每个端口独立冲突域，无碰撞。
   - **带宽**：
     - 集线器：共享总带宽，N 台设备每台平均带宽为总带宽的 1/N。
     - 交换机：每个端口独享带宽，全双工下可同时发送接收。
   - **通信模式**：
     - 集线器：半双工，同一时刻只能一个站发送。
     - 交换机：全双工（多数型号），可同时多对通信。
   - **并行性**：
     - 集线器：无并行能力。
     - 交换机：支持多个端口对同时通信，总吞吐量远高于集线器。

## 7. 内部网关协议（RIP 与 OSPF）及外部网关协议（BGP）对比



1. **RIP 协议特点**
   - 属于**距离向量**协议。
   - 度量标准为**跳数**，最大 15 跳，16 跳表示不可达。
   - 每隔 30 秒向相邻路由器交换**整个路由表**。
   - 收敛速度慢，可能出现“坏消息传播慢”的问题。
   - 适用于小型网络，实现简单，开销小。

2. **OSPF 协议特点**
   - 属于**链路状态**协议。
   - 使用 Dijkstra 最短路径算法（SPF）。
   - 仅当链路状态发生变化时，才用**洪泛法**发送更新信息。
   - 收敛速度快（毫秒到秒级），无“计数到无穷”问题。
   - 支持划分**区域**，主干区域为 `0.0.0.0`，减少路由信息量。
   - 适用于大型复杂网络。

3. **RIP 与 OSPF 对比总结**
   - 算法：RIP 距离向量，OSPF 链路状态。
   - 度量：RIP 跳数，OSPF 代价（带宽、时延等综合）。
   - 交换信息：RIP 周期交换整表，OSPF 变化时才交换链路状态。
   - 收敛速度：RIP 慢（可能数分钟），OSPF 快（<100ms）。
   - 规模：RIP 适小，OSPF 适大。

4. **BGP 协议特点（外部网关协议）**
   - 用于不同自治系统（AS）之间交换路由信息。
   - 不是寻找最优路径，而是基于**策略**（如政治、经济、安全）选择可达路由。
   - 交换信息为**网络前缀（CIDR）**及其经过的 AS 路径（AS_PATH），防止环路。
   - 使用 TCP 179 端口，可靠传输。
   - 支持 CIDR 和路由聚合。

## 8. IPv4 与 IPv6 的主要区别及过渡技术



1. **地址长度与表示法**
   - IPv4：32 位，点分十进制，如 `192.168.1.1`，约 43 亿地址。
   - IPv6：128 位，冒号十六进制，如 `2001:db8::1`，地址空间极大。

2. **首部结构**
   - IPv4 首部：20~60 字节（可变长度），包含 IHL、总长度、标识、标志、片偏移、TTL、协议、首部检验和等。
   - IPv6 首部：固定 40 字节，字段减少为 8 个，取消首部检验和、分片相关字段（分片由扩展首部实现），TTL 改称为“跳数限制”。
   - IPv6 使用扩展首部代替 IPv4 的选项字段，提高了处理效率。

3. **地址配置**
   - IPv4：手动配置、DHCP（动态主机配置协议）。
   - IPv6：支持无状态自动配置（SLAAC），结合邻居发现协议（NDP）；也支持 DHCPv6。

4. **安全与协议**
   - IPv4：IPSec 可选，需单独部署。
   - IPv6：IPSec 强制支持（但非必须使用），NDP 替代 ARP，ICMPv6 集成更多功能。

5. **过渡技术**
   - **双协议栈（Dual Stack）**：设备同时运行 IPv4 和 IPv6 协议栈，根据目的地址选择使用。
   - **隧道技术（Tunneling）**：将 IPv6 数据报封装在 IPv4 数据报中穿越 IPv4 网络，常用隧道协议有 6to4、Teredo、ISATAP、GRE 等。
   - **翻译技术（NAT64/DNS64）**：将 IPv6 地址映射为 IPv4 地址，实现纯 IPv6 节点与纯 IPv4 节点通信。

## 9. IP 地址与 MAC 地址的区别、联系及 ARP 解析过程



1. **IP 地址**
   - 工作层次：网络层。
   - 长度：32 位（IPv4），128 位（IPv6）。
   - 性质：逻辑地址，可配置，可变化。
   - 作用：标识主机在网络中的位置，用于跨网络寻址和路由。
   - 分配：由机构（ICANN、ISP）分配或 DHCP 动态获取。

2. **MAC 地址**
   - 工作层次：数据链路层。
   - 长度：48 位，通常表示为十六进制（如 `00:1A:2B:3C:4D:5E`）。
   - 性质：物理地址，固化在网卡 ROM 中，全球唯一（理论上）。
   - 作用：同一链路上标识网卡，用于帧的交付。
   - 分配：由 IEEE 管理，厂家生产时烧录。

3. **两者的联系**
   - IP 数据报必须封装成帧才能发送，帧必须有目的 MAC 地址。
   - 在同一网络中，通过 ARP 协议将目的 IP 地址解析为 MAC 地址。
   - 跨网络时，IP 数据报逐跳转发，每跳都需要将下一跳 IP 解析为下一跳 MAC。

4. **同一局域网 ARP 解析过程**
   - ① 主机 A 检查 ARP 缓存，若有主机 B 的 IP→MAC 映射则直接使用。
   - ② 若无，A 广播 ARP 请求帧（目的 MAC = FF-FF-FF-FF-FF-FF），内容：“谁是 192.168.1.5？告诉 192.168.1.1（MAC：AA:AA:AA:AA:AA:AA）”。
   - ③ 所有主机接收，只有 B 匹配 IP，B 单播回复 ARP 响应：“我是 192.168.1.5，我的 MAC 是 BB:BB:BB:BB:BB:BB”。
   - ④ A 将映射存入缓存（TTL 通常 10~20 分钟），然后发送数据帧。

5. **不同网络通信时 ARP 的使用**
   - 主机 A 发现目的 IP 不在同一网络，则确定默认网关 IP。
   - A 使用 ARP 解析**网关的 MAC 地址**（过程同上）。
   - 数据帧发往网关，网关再重复类似过程转发。

## 10. TCP 可靠传输机制（滑动窗口、确认、重传、拥塞控制）



1. **序号与确认**
   - TCP 将字节流中的每个字节编号（序号）。
   - 确认号字段表示期望收到的下一个字节序号，即累积确认。
   - 接收方可以发送单独的 ACK 或捎带确认。

2. **滑动窗口**
   - 发送方维护一个发送窗口，窗口内的数据可以连续发送，无需等待确认。
   - 接收方通告一个接收窗口（rwnd），告诉发送方还能接收多少字节。
   - 窗口大小动态变化，实现流量控制。
   - 发送窗口移动：收到 ACK 后，窗口向前滑动。

3. **超时重传**
   - 每个发送的报文段启动重传计时器，超时未收到 ACK 则重传。
   - 超时时间（RTO）基于往返时间（RTT）自适应计算，使用 Karn 算法避免歧义。

4. **拥塞控制**
   - 目的：防止网络过载导致大量丢包。
   - **慢启动**：发送窗口从 1 个 MSS 开始，每收到一个 ACK 加倍，指数增长直到慢启动阈值（ssthresh）。
   - **拥塞避免**：达到阈值后，每收到一个 ACK 增加 1 个 MSS，线性增长。
   - **快重传**：发送方收到三个重复 ACK 时，立即重传丢失的报文段（不必等到超时）。
   - **快恢复**：快重传后，将 ssthresh 设为当前窗口的一半，窗口设为 ssthresh，进入拥塞避免。

5. **停止等待与连续 ARQ 的区别**
   - **停止等待协议**：发送一个分组，等待确认后再发下一个。优点简单，缺点信道利用率低（尤其 RTT 大时）。
   - **连续 ARQ（回退 N）**：连续发送窗口内多个分组，使用累积确认。若某一分组丢失，重传该分组及其之后所有分组（Go-back-N）。效率高于停止等待。
   - **选择重传（Selective Repeat）**：只重传丢失的分组，需要 SACK 选项支持，效率最高。

## 11. OSI、TCP/IP、五层模型比较及各层主要功能



1. **OSI 七层模型（从下至上）**
   - **物理层**：传输比特流，定义电气、机械、过程特性。
   - **数据链路层**：封装成帧，透明传输，差错控制（CRC），MAC 地址。
   - **网络层**：路由选择，分组转发，IP 协议。
   - **运输层**：端到端通信，可靠传输（TCP）与不可靠（UDP）。
   - **会话层**：建立、管理、终止会话（OSI 独有）。
   - **表示层**：数据格式转换、加密、压缩（OSI 独有）。
   - **应用层**：提供网络应用服务（HTTP、FTP、SMTP 等）。

2. **TCP/IP 四层模型**
   - **网络接口层**：未具体定义，对应物理层和数据链路层。
   - **网际层**：IP 协议，路由选择，分组转发。
   - **运输层**：TCP 和 UDP。
   - **应用层**：包含 OSI 的应用层、表示层、会话层功能。

3. **五层混合模型（教学常用）**
   - **物理层**：同 OSI。
   - **数据链路层**：同 OSI。
   - **网络层**：同 OSI。
   - **运输层**：同 OSI。
   - **应用层**：合并了 OSI 的会话、表示、应用层，主要对应 TCP/IP 的应用层。

4. **各层核心功能总结**
   - 物理层：比特流传输，定义介质接口。
   - 数据链路层：帧封装，MAC 寻址，差错检测。
   - 网络层：IP 寻址，路由选择，分组转发。
   - 运输层：端口多路复用/分解，可靠传输（TCP）或不可靠（UDP）。
   - 应用层：用户进程通信，提供各种网络应用协议。

## 12. DHCP 完整工作过程（DORA + 租约更新）



1. **发现阶段（Discover）**
   - DHCP 客户端启动时，广播发送 **DHCPDISCOVER** 报文。
   - 源 IP：`0.0.0.0`，目的 IP：`255.255.255.255`，UDP 源端口 68，目的端口 67。
   - 目的：寻找网络中的 DHCP 服务器。

2. **提供阶段（Offer）**
   - 所有收到 Discover 的 DHCP 服务器（如果有）从地址池中预选一个 IP 地址。
   - 服务器单播（或广播）返回 **DHCPOFFER** 报文，包含提供的 IP、子网掩码、租用期、默认网关、DNS 等。
   - 客户端可能收到多个 Offer，通常选择第一个。

3. **请求阶段（Request）**
   - 客户端广播发送 **DHCPREQUEST** 报文，明确告知选中的服务器：“我接受你提供的 IP”。
   - 同时隐式告知其他服务器：“我不使用你们的 IP”。
   - 报文中包含选中的服务器标识和请求的 IP。

4. **确认阶段（Acknowledge）**
   - 被选中的服务器收到 Request 后，若 IP 仍可用，则返回 **DHCPACK** 报文，正式确认分配。
   - 客户端收到 ACK 后，开始使用该 IP，并设置两个计时器：T1 = 0.5 × 租用期，T2 = 0.875 × 租用期。
   - 若服务器无法满足（如 IP 已被占用），则返回 **DHCPNACK**，客户端需重新开始 Discover。

5. **租约更新过程**
   - **T1 时刻（租期过半）**：客户端单播发送 DHCPREQUEST 给原服务器，请求续约。
     - 若收到 DHCPACK，重置计时器。
     - 若收到 DHCPNACK，立即停止使用 IP，重新 Discover。
   - **T2 时刻（租期 87.5%）**：若 T1 时未收到响应（或续约失败），客户端广播 DHCPREQUEST 续约。
   - **租期结束**：若始终未收到应答，IP 地址失效，客户端停止使用，重新 Discover。

6. **主动释放**
   - 客户端可随时发送 **DHCPRELEASE** 报文，主动释放 IP 地址，服务器回收该地址。

## 13. PPP 协议的组成、帧格式及透明传输方法



1. **PPP 协议的三个组成部分**
   - **封装方法**：将 IP 数据报（或其他网络层协议数据）封装到串行链路的帧中。
   - **链路控制协议（LCP）**：负责建立、配置、测试和终止数据链路连接，协商最大接收单元（MRU）、认证协议（如 PAP、CHAP）等。
   - **网络控制协议（NCP）**：针对不同网络层协议进行配置，例如 IPCP 用于分配 IP 地址、DNS 等。

2. **PPP 帧格式（从开始到结束）**
   - **标志字段 F**：1 字节，固定为 `0x7E`，表示帧的开始和结束。
   - **地址字段 A**：1 字节，固定为 `0xFF`（PPP 中无意义，但保留）。
   - **控制字段 C**：1 字节，固定为 `0x03`。
   - **协议字段**：2 字节，标识信息字段的类型，常见值：
     - `0x0021`：IP 数据报
     - `0xC021`：LCP 数据
     - `0x8021`：NCP 数据
   - **信息字段**：变长，最大不超过 MRU（默认 1500 字节），存放网络层 PDU。
   - **填充字段**：可选，使帧长度满足要求（某些实现需要）。
   - **帧检验序列 FCS**：2 或 4 字节，CRC 校验。
   - **标志字段 F**：1 字节，`0x7E` 结束。

3. **异步传输下的透明传输（字符填充）**
   - 将信息字段中出现的 `0x7E`（标志）转换为两字节序列 `0x7D`、`0x5E`。
   - 将出现的 `0x7D`（转义字符）转换为 `0x7D`、`0x5D`。
   - 将控制字符（ASCII 值小于 `0x20`）前面加上 `0x7D`，并将该字符编码加 `0x20`。例如 `0x03` 转换为 `0x7D`、`0x23`。
   - 接收端执行逆变换还原。

4. **同步传输下的透明传输（比特填充）**
   - 发送端在连续 5 个 1 之后插入一个 0。
   - 接收端在连续 5 个 1 之后删除一个 0。
   - 这样保证帧内不会出现 `0x7E`（即 `01111110`）的误判。

## 14. CDMA 码分多址原理与正交码片序列计算



1. **CDMA 工作原理**
   - 每个站被分配一个唯一的 m 比特码片序列（chip sequence），通常用 `+1` 表示比特 1，`-1` 表示比特 0。
   - 发送比特 1 时，发送原码片序列；发送比特 0 时，发送码片序列的反码（所有位取反）。
   - 不同站的码片序列**正交**，即规格化内积为 0。
   - 所有站同时发送，信号在空中叠加。
   - 接收端若要接收站 S 的数据，将叠加信号与 S 的码片序列做内积（除以 m），结果：
     - 若为 `+1`，则 S 发比特 1。
     - 若为 `-1`，则 S 发比特 0。
     - 若为 `0`，则 S 未发送。

2. **已知条件**
   - 叠加序列：`(-1, +1, -3, +1, -1, -3, +1, +1)`
   - 各站码片序列：
     - A: `(-1,-1,-1,+1,+1,-1,+1,+1)`
     - B: `(-1,-1,+1,-1,+1,+1,+1,-1)`
     - C: `(-1,+1,-1,+1,+1,+1,-1,-1)`
     - D: `(-1,+1,-1,-1,-1,-1,+1,-1)`

3. **计算 A 的内积**
   - 乘积：`(-1)×(-1)=1`，`(+1)×(-1)=-1`，`(-3)×(-1)=3`，`(+1)×(+1)=1`，`(-1)×(+1)=-1`，`(-3)×(-1)=3`，`(+1)×(+1)=1`，`(+1)×(+1)=1`
   - 求和：`1-1+3+1-1+3+1+1 = 8`
   - 内积 = `8/8 = 1` → **A 发 1**

4. **计算 B 的内积**
   - 乘积：`(-1)×(-1)=1`，`(+1)×(-1)=-1`，`(-3)×(+1)=-3`，`(+1)×(-1)=-1`，`(-1)×(+1)=-1`，`(-3)×(+1)=-3`，`(+1)×(+1)=1`，`(+1)×(-1)=-1`
   - 求和：`1 -1 -3 -1 -1 -3 +1 -1 = -8`
   - 内积 = `-8/8 = -1` → **B 发 0**

5. **计算 C 的内积**
   - 乘积：`(-1)×(-1)=1`，`(+1)×(+1)=1`，`(-3)×(-1)=3`，`(+1)×(+1)=1`，`(-1)×(+1)=-1`，`(-3)×(+1)=-3`，`(+1)×(-1)=-1`，`(+1)×(-1)=-1`
   - 求和：`1+1+3+1-1-3-1-1 = 0`
   - 内积 = `0/8 = 0` → **C 未发送**

6. **计算 D 的内积**
   - 乘积：`(-1)×(-1)=1`，`(+1)×(+1)=1`，`(-3)×(-1)=3`，`(+1)×(-1)=-1`，`(-1)×(-1)=1`，`(-3)×(-1)=3`，`(+1)×(+1)=1`，`(+1)×(-1)=-1`
   - 求和：`1+1+3-1+1+3+1-1 = 8`
   - 内积 = `8/8 = 1` → **D 发 1**

7. **结论**
   - A 发送 1，B 发送 0，C 未发送，D 发送 1。

## 15. 综合应用题：文件传输中的协议栈交互（FTP）



1. **应用层**
   - 用户使用 FTP 客户端（应用进程）与 FTP 服务器建立**控制连接**。
   - 控制连接使用 TCP 熟知端口 21，用于传输用户名、密码、命令（如 `RETR filename`）和响应。
   - 当用户请求下载文件时，服务器创建**数据连接**（TCP 熟知端口 20），用于传输文件内容。
   - FTP 数据连接传输完成后关闭，控制连接可继续使用。

2. **运输层**
   - 控制连接和数据连接均使用 **TCP** 协议。
   - TCP 提供可靠、面向连接的服务，确保文件数据不丢失、不重复、按序到达。
   - 每个连接需经过**三次握手**建立，结束时**四次挥手**释放。
   - TCP 使用序号、确认、滑动窗口、超时重传等机制保证可靠性。

3. **网络层**
   - TCP 报文段被封装成 **IP 数据报**。
   - 源主机根据目的 IP 地址（FTP 服务器 IP）查找路由表，确定下一跳。
   - 路由器逐跳转发 IP 数据报，每跳根据 IP 地址查找路由表（由 RIP、OSPF 等协议生成）。
   - 可能经过多个路由器，IP 头部 TTL 每跳减 1，防止循环。

4. **数据链路层**
   - 在每一段链路上，IP 数据报封装成帧（如以太网帧）。
   - 发送方需要知道下一跳的 MAC 地址，使用 **ARP 协议**解析。
   - 帧尾部添加 FCS（CRC 校验），接收方检测错误，出错则丢弃，由 TCP 重传恢复。

5. **物理层**
   - 帧转换为比特流，经双绞线、光纤或无线介质传输。
   - 物理层定义了电气特性、电缆接口、传输速率等。

6. **TCP 如何确保文件完整**
   - **校验和**：检测数据是否损坏，损坏则丢弃，接收方不确认，发送方超时重传。
   - **序号与确认**：每个字节编号，接收方发送 ACK 确认已收到的序号，发送方只重传未确认的部分。
   - **超时重传**：发送方为每个报文段设置计时器，超时未收到 ACK 则重传。
   - **滑动窗口与流量控制**：接收方通告窗口大小，防止发送方发送过快导致接收方缓存溢出。
   - **拥塞控制**：慢启动、拥塞避免、快重传、快恢复，防止网络过载导致丢包。
   - 这些机制共同保证了即使下层不可靠（IP 只提供尽最大努力交付），TCP 仍能提供可靠的字节流服务。
